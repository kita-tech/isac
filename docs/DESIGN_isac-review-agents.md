# 設計書: isac-review エージェントチームモード（`--agents`）

> ステータス: **設計確定（#1〜#5 すべて決定済）** / 実装前 / 次は最小 PoC（§10.4）で接地
> 目的: **アンカリング・収束バイアスの低減**に絞った、ブラインド並列エージェント（Workflow 実装）による設計レビュー方式。
> 実装手段: **Claude Code Workflow スクリプト**（§10・未決#3で決定）。

## 1. 目的とスコープ

### 対象にするバイアス（唯一）
- **アンカリング/収束/自己一貫性バイアス**: 1体の Claude が4ペルソナを順に生成するため、後続ペルソナが前の意見に同調し、投票が「浮かびかけた合意の追認」になる問題。

### 明示的に対象外（誤解防止のため必ず明記）
- **相関した知識の盲点（plausible-but-wrong / 事実バグ）**: 全エージェントが同一モデル（Claude）である限り解消しない。→ **実行検証ゲート（PR#63、導入済）** と **別モデルのクロスチェック** の担当。本モードの効能ではない。
- 本モードは「同調を減らす」機能であり「事実正しさの保証」ではない。SKILL.md/CLAUDE.md にこの線引きを明記する。

## 2. 現状と課題
- 現行 `isac-review`: 1コンテキスト内で4ペルソナを逐次生成 → 構造的にアンカリング。
- `isac-autopilot`: Phase4 レビューは「1体の別エージェント」に委譲（バイアス低減の実績あり）。ただし *N体の別ペルソナ* ではない。
- → 「ペルソナごとに独立エージェント（ブラインド）」はまだ存在しない。本設計で新設する。

## 3. 設計方針
1. **ブラインド並列ラウンド**: 各ペルソナを互いの出力を見せずに別コンテキストで起動。コンテキスト隔離＝アンカリング除去（プロンプトで「独立に」と頼むより確実）。
2. **種（seed）の多様化**: 単なるN体コピーではなく、役割・立場・インセンティブを各体で変える。結論を実際に脱相関させる本命レバー。
3. **機械的合成**: 集計は投票を数え、異論は逐語で列挙。→ 案B（Workflow）では **JS コードで決定論的に実施**（バイアスの余地なし）。
4. **opt-in**: 既定は現行インライン方式（安価）。`--agents` で本モード。重い意思決定のみ。

## 4. アーキテクチャ（フェーズ）

```
[/isac-review --agents]  →  main loop が Workflow を起動（Skill が指示＝明示オプトイン）
  │  args: {議題, 設計文, N, 多様化した役割定義, blindOnly}
  │
  Workflow スクリプト（JS が決定論的にオーケストレート）:
  ├─▶ Round 1（parallel）: N体の reviewer agent を並列起動（互いに不可視＝ブラインド）
  │      各 agent: 自分のレンズで意見。runnable な主張は bash/docker で自己検証(#63)
  │      agent({schema}) で構造化結果を検証付き回収
  ├─▶ [既定ON] Round 2（parallel・再spawn）: 各体の Round1＋他体の匿名意見を注入し新規N体を起動
  │      out: {revised_position, change_justification|null, refutations[], new_risks[]}
  └─▶ 合成（★JS のみ・エージェント0体）: 投票集計・争点の逐語束ね・立場変更ログ・検証ラベル集約
         → **推奨レポート**（§5.3、decision ではない）。割れたら裁定せずユーザーへエスカレーション
                │
  [main loop] ──┴─▶ 推奨レポートを提示。**Memory Service への decision 自動保存はしない**（§11）
                     決定として残すかは人間が /isac-decide で確認・確定
```

## 5. エージェント構成と体数

### 役割ごとの体数（案B確定後）
| 役割 | 体数 | 備考 |
|------|------|------|
| **Reviewer（Round1・ブラインド）** | N | ペルソナ数。既存モード（2/4/5/10）に対応 |
| **Reviewer（Round2・再spawn）** | +N（debate時） | Workflow は同一体継続不可 → 新規N体に前ラウンドを注入。**2N spawn を許容**（コスト問題が出たら再考） |
| Synthesizer（合成） | **0** | 集計は Workflow の JS で決定論実施（§5.1）。エージェント不要 |

→ **実効 spawn ≒ 2N（debate）／ N（`--blind-only`）**。役割数は N。

### 5.1 合成の担当【決定 — 未決#1】

**原則: 合成役のバイアスは「担当の選択」ではなく「裁定を機械化・投票化」して消す。**

工程を2つに分ける:
- **集計（aggregation）**: 投票を数え、異論を逐語で列挙、検証ラベル（✅/⚠️）を束ねる。
- **裁定（adjudication）**: 独自の結論を出さない。**結論 = Reviewer の投票結果そのもの**。割れたら**裁定せずユーザーにエスカレーション**。少数意見は minority report で保持。

**案B（Workflow）採用による強化**: 集計・裁定ロジックは **workflow スクリプトの JS で決定論的に実装**（票を数える／争点を逐語で並べる／閾値割れをif分岐でエスカレーション）。
→ **合成にエージェントを使わない＝合成役のバイアスが原理的にゼロ**。
→ §5.1 で当初検討した「main loop 合成 vs 専用エージェント合成（利益相反対策）」の A/B **問題自体が消える**。JS 集計は authoring Claude の判断を通さないため、autopilot 文脈でも利益相反が生じない（**+1体も不要**）。

**実装要点**: 「機械集計」は JS が担保するので LLM の editorialize リスクが無い。各 Reviewer の構造化出力（§5.2 のスキーマ）を JS が集計するだけ。

### 5.2 ラウンド構成【決定 — 未決#2、案B反映で更新】

**決定**: 既定 **Round1（ブラインド）→ Round2（構造化相互反論・1回）→ JS合成**。
`--blind-only` で Round2 省略。**Round2 は最大1回**（同一モデル多回転は herding 増＆逓減）。

| ラウンド | 内容 | 入出力（agent の schema） |
|----------|------|--------|
| **Round1 ブラインド** | `parallel()` でN体を独立起動。runnable な主張は自己で実行検証(#63) | out: `{position, rationale, concerns[], recommendation, verified_claims[]}` |
| **Round2 相互反論**（既定ON） | 各体の Round1＋**他体の匿名意見**を注入し、**新規N体を再spawn**（§下）。単一パス | in: 自Round1＋匿名他意見 / out: `{revised_position, change_justification\|null, refutations[], new_risks[]}` |
| **合成** | JS（§5.1） | 最終レポート（§5.3） |

**Round2 のアンカリング抑制策（heuristics。§8 の A/B で実証する前提）**:
1. **匿名化**: Round1 意見を肩書・人物を伏せ「意見A/B/C」で提示 → 権威同調の除去。
2. **単一パス同時**: 全体が一斉に1回だけ批判。ターン制往復なし。
3. **変更には理由必須**: 立場変更時は動かした具体的論拠を明記（`change_justification`）。無根拠な同調を可視化・抑止。
4. **懐疑役は据え置き**: Round2 でも異論インセンティブ保持。
5. **Round1 の立場を保存**: 討論が何を変えたか監査可能に（少数意見の沈黙の消去を防ぐ）。

**Round2 の実装＝再spawn（案B確定で SendMessage 継続から変更）**:
- Workflow の `agent()` は毎回ステートレス起動で、**スクリプト内での同一エージェント継続（SendMessage 相当）は非対応**（§10・裏取りで確認）。
- よって Round2 は「新規N体を起動し、各体に自分の Round1 出力＋他体の匿名意見を**プロンプト注入**」する。
- 結果: debate 時 spawn は **2N**（許容）。むしろ再spawn の方がブラインド性は安全（自分の思考トレースへの過剰固執を避けられる）。

### 5.3 出力アーティファクト（成果物スキーマ）

「しっかりした出力」を保証するため最終レポート構造を固定。JS が Reviewer の構造化出力から機械生成する:

| セクション | 中身 | 質の担保 |
|-----------|------|----------|
| **投票結果** | 選択肢別票数。**結論＝これ** | 裁定バイアス排除 |
| 合意点 | 全体一致点 | — |
| **争点** | 対立点。各サイドの論拠を**逐語** | 異論の平滑化禁止 |
| 少数意見（minority report） | 却下されなかった異論 | 沈黙の消去防止 |
| **立場変更ログ** | 誰が／どの論拠で／なぜ R1→R2 で変えたか | 討論の付加価値＋herding 監査 |
| 実行検証サマリ | ✅検証済 / ⚠️未検証 の主張一覧（#63） | 事実バグ対策との連結 |
| 未解決・要ユーザー判断 | 割れて裁定不能な点 | 無理な裁定をしない |

- **この最終レポートは「推奨（recommendation）」であって「決定（decision）」ではない**（§11）。Memory Service への decision 自動保存はしない。

### N の推奨と体数
- **下限3 / スイートスポット4〜6 / 既定4**。同一モデルゆえ N を増やすほど相関重複が増え限界効用逓減（10体でも“10人の独立専門家”にはならない）。
- **並列上限**: `min(16, cores-2)`。8コア機＝**6**。各ラウンド内で N≤6 は1波、N>6 は波分割（ブラインド性は同時実行不要なので波でも成立）。

| モード | Reviewer(役割) | Round1 | Round2(debate) | 合成 | 総spawn | 8コア |
|--------|------|--------|--------|------|---------|-------|
| `--agents --quick` | 2 | 2 | 2 | 0(JS) | 4 | 各1波 |
| `--agents`（既定） | **4** | 4 | 4 | 0(JS) | **8** | 各1波 |
| `--agents --full` | 5 | 5 | 5 | 0(JS) | 10 | 各1波(6まで) |
| `--agents --team` | 10 | 10 | 10 | 0(JS) | 20 | 各2波 |
| `--agents --blind-only` | N | N | — | 0(JS) | N | — |

## 6. 種（seed）の多様化 設計
「あなたはペルソナX」ではなく**脱相関する軸**を割り当てる:
- **立場**: 推進派（採用を弁護）／反対派（致命傷を見つければ報われる、と明示インセンティブ）
- **リスク許容度**: 保守的／挑戦的
- **優先軸**: 速度／堅牢性／運用コスト
- **必須**: 懐疑的レビュアー1体（plausible-but-wrong ハンター、実行検証を要求）
- 例（既定4体）: ①推進×アーキ ②反対×運用 ③中立×実装 ④懐疑（検証駆動）
- 実装: 役割定義を Workflow の `args` で渡し、各 `agent()` プロンプトに差し込む。

## 7. 制約・リスク
- **コスト/レイテンシ**: debate 時 2N。10体なら 20 spawn。→ opt-in 必須。コスト問題顕在化時に `--blind-only` 既定化等を再検討（現時点では 2N 許容）。
- **MCP**: Workflow エージェントは **ToolSearch 経由で session 接続済み MCP に到達可**（＝従来の「サブエージェント MCP 不可」より緩い）。ただし headless/cron 実行では対話認証系 MCP が欠落しうる。設計レビューが提供テキスト中心なら影響小。
- **合成の再収束**: JS 集計で原理的に排除（案B の利点）。
- **相関盲点は残る**（対象外・再掲）。
- **Round2 のアンカリング復活**: トレードオフ。既定ON、`--blind-only` で回避可。§8 で効果を実測して既定を再判断。

## 10. 実装手段（Workflow）【決定 — 未決#3】

**決定**: **案B = Claude Code Workflow スクリプト**で実装。`/isac-review --agents` から main loop が Workflow を起動。

### 10.1 裏取り結果（claude-code-guide × Workflow 一次仕様の突き合わせ）
> ⚠️ **provenance**: claude-code-guide の回答に **plausible-but-wrong が2点**あり、Workflow ツール一次仕様との照合で訂正した（本設計のクロスチェック原則の実践）。

| 論点 | guide | 一次仕様（採用） |
|------|-------|------------------|
| 構造化出力 | 「無い、JS 自前」 | **`agent(prompt,{schema})` が検証済みオブジェクト返却＋不一致自動リトライ。組み込みで有り** |
| 同一体継続 | 「SendMessage で可能かも」 | **script 内では非対応 → Round2 は再spawn** |
| Skill→Workflow | 「不明」 | **可（skill/slash command が指示する Workflow は明示オプトインに該当）** |
| 並列/ブラインド | 「最大16、ブラインド」 | 上限 `min(16,cores-2)`＝8コアで6。並列体は相互不可視 |
| Bash/MCP | エージェントは可 | **可**（Reviewer が実行検証を回せる。MCP は ToolSearch 経由） |
| 実行モデル | バックグラウンド＋通知 | 一致。`resumeFromRunId` で再開可 |

### 10.2 実装形態
- **配置**: `.claude/workflows/isac-review-agents.js`（named workflow）。`meta` に name/description/phases。
- **起動**: SKILL.md（`isac-review`）に「`--agents` 指定時は Workflow を起動」フローを記述。`args` で {議題, 設計文, N, roles[], blindOnly} を渡す。
- **構造化**: 各ラウンド `agent(prompt, {schema})` でスキーマ検証。
- **並列**: Round1/Round2 とも `parallel()`。ブラインドは並列内で自動担保。
- **合成**: workflow スクリプトの JS（集計・逐語・エスカレーション）。エージェント0体。

### 10.3 ISAC 配布経路の新設（案B固有タスク・接地で確認）
- 現状: `bin/isac` は hooks/skills/agents のみ配布。**workflows は未対応**（`.claude/workflows/` はどこにも無い＝確認済み）。
- 追加: `cmd_install`/`cmd_update` に `.claude/workflows/` → `~/.isac/workflows/` のコピー、`cmd_init` に各プロジェクトへの symlink（skills と同型）。
- 難度は低いが**ゼロではない実装タスク**として PR に含める。

### 10.4 PoC で潰す残リスク
- Round2 再spawn の**注入プロンプト設計**（自Round1＋匿名他意見を渡して正しく再評価させる）。
- named workflow の **ISAC 配布の手触り**（symlink 済み workflow が別プロジェクトで `/` 起動できるか）。
- → 4体・1議題の最小 PoC で先に確認してから本実装。

## 11. ガバナンス: 推奨と決定の分離【決定 — 未決#4「強」】

**原則: AI は推奨する、決定は人間が下す。**（ISAC 既存原則「AI による PR のマージ禁止＝マージは人間が判断」と同型）

### 決定事項
1. **agents モードは Memory Service に decision を自動保存しない。** 出力は **推奨レポート（recommendation）** に留まる。
   - 理由: 「レビューの結論（＝投票、§5.1）」は *AIパネルの推奨* であって *チームが commit した決定* ではない。`type=decision` で保存すると、AI 推奨が「決定事項」として自動注入され、**人間が決めていないものが権威を持って流通**する（＝ガバナンス欠落）。
2. **決定として残すかは人間が明示的に行う** → 既存 `/isac-decide` を使用。
   - 推奨レポートを**下書きとして流し込む**のは可。ただし **人間が決定文を確定して初めて `type=decision`**（「Y/N 承認」ではなく「決定文の確定＝著者は人間」）。
   - ＝ **記憶に "決定" として出るものは、必ず人間が著者**。パネルは材料提供に徹する。
3. **想起時の権威分離**: 推奨は「📋 プロジェクト決定事項」として注入されない（そもそも保存しない）。決定は人間が `/isac-decide` した時のみ、決定事項として想起される。

### この原則の波及（任意・別タスク）
- **現行インライン `isac-review` も同じ穴**（`type=decision` を Y/N で自動保存している）。本原則を CLAUDE.md に一段書けば、全レビュー系スキルに適用でき一貫する。→ 別 PR 候補。

### 推奨レポートのアーカイブ（任意）
- レポート本体はセッション出力。残したければ PR コメントや md に貼る（Memory Service ではない）。
- どうしても記憶に痕跡を残したい場合のみ、**非権威**の `type=work`（「レビュー実施記録」＝“最近の作業”として想起、決定事項扱いにならない）で要点のみ。既定はやらない。

## 12. CLI フラグ仕様【決定 — 未決#5】

### 3つの直交軸
| 軸 | 決めるもの | フラグ | 既定 |
|----|-----------|--------|------|
| ① 体数 N | レビュアー数 | `--quick`(2) / 無(4) / `--full`(5) / `--team`(10) | 4 |
| ② 実行方式 | インライン（1体が演じる）か 別エージェント | `--agents` | インライン |
| ③ ラウンド | 討論あり/なし（②が前提） | `--blind-only`（Round2省略） | 討論あり |

3軸は独立。掛け合わせ可能。

### 組合せ例
| コマンド | 方式 | 体数 | ラウンド | 総spawn(8コア) |
|----------|------|------|----------|------|
| `/isac-review` | インライン | 4 | — | 0（現行のまま） |
| `/isac-review --full` | インライン | 5 | — | 0（現行のまま） |
| `/isac-review --agents` | 別 | 4 | 討論 | 8 |
| `/isac-review --agents --quick` | 別 | 2 | 討論 | 4 |
| `/isac-review --agents --full` | 別 | 5 | 討論 | 10 |
| `/isac-review --agents --team` | 別 | 10 | 討論 | 20（各R 2波） |
| `/isac-review --agents --blind-only` | 別 | 4 | 無 | 4 |
| `/isac-review --agents --full --blind-only` | 別 | 5 | 無 | 5 |

### バリデーション規則
1. **体数フラグは相互排他**: `--quick/--full/--team` の同時指定はエラー（「体数フラグは1つ」）。
2. **`--blind-only` は `--agents` 前提**: 単独指定は `--agents --blind-only` と自動解釈し、その旨を一言告知。
3. **後方互換**: `--agents` 無しは現行と完全に同一挙動（インライン。体数フラグの意味も不変）。
4. **既定値**: `--agents` 単独 = 体数4・討論あり。

### 役割割当のスケール（§6 と連結、懐疑役は必須）
| N | 役割セット |
|---|-----------|
| 2 (`--quick`) | 推進 / **懐疑** |
| 4 (既定) | 推進×アーキ / 反対×運用 / 中立×実装 / **懐疑**(検証駆動) |
| 5 (`--full`) | +セキュリティ or perf レンズ1 |
| 10 (`--team`) | 立場×ドメインを広く配分。※同一モデル相関で逓減する旨を出力に注記 |

### コストガード
- **総 spawn > 12 の組合せは実行前に確認**（例: `--agents --team` = 20 spawn → 「N体×2ラウンド＝20 spawn を起動します。続けますか？」）。opt-in の二重ゲート。
- 実行方式名は **`--agents`**（`--panel` 等には改名しない。§6/§11 と呼称統一）。
